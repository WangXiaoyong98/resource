# 大疆三面

1、Uboot 中不同异常向量表的设置；
U-Boot中异常向量表的设置，本质上是软件（U-Boot）遵循硬件（CPU架构）的规则，在内存的特定位置布置好一张"跳转地图"。当CPU遇到异常时，它会自动到这张地图上找到对应的入口，执行处理代码。
这个设置过程主要包含建立向量表、填充入口地址和安装向量表三个关键步骤。
（1）1. 建立向量表：一张固定的"跳转地图"
（2）2. 填充入口地址：把"处理函数"的地址填到表后面
（3） 安装向量表：告诉CPU表在哪里（ARM64）

2、如何设置 Uboot 的 C 语言环境；
核心四步：
（1）、设置栈指针 ： 有了 sp ，C函数才能调用，局部变量才能存活
（2）、清零BSS段 ： C标准规定未初始化的全局静态变量值必须清0，不清零会导致随机值，程序逻辑出错
（3）、初始化数据段 
（4）、跳转到main函数
两个重要的C阶段，board_init_f 初始化C环境，board_init_r 跳转到main函数

board_init_f (前初始化)：此时 U-Boot 可能还在只读存储器（如Flash）中执行。这个函数主要负责两件事：
基础硬件初始化：比如初始化串口（这样就能打印信息了）、定时器等。
规划内存布局：计算出代码段、数据段、堆、栈最终在内存（RAM）中的位置，为“重定位”做准备。
重定位 (Relocation)：board_init_f 结束后，U-Boot 会把自身从慢速的 Flash 拷贝到高速的 RAM 中。这是一个关键动作，使得后续执行速度大大加快。

board_init_r (后初始化)：此时 U-Boot 已运行在 RAM 中。这个函数是“大管家”，负责完成所有剩下的初始化工作，比如初始化网卡、MMC、命令行接口等，最终进入我们常见的命令行提示符，等待用户输入命令。

4、Uboot 启动的流程；
第一阶段：汇编阶段（CPU刚上电时）：这个阶段与硬件高度相关，主要用汇编语言完成，因为此时 C 环境还没准备好。
    1. 设置异常向量表
    作用：CPU 上电后，第一件事就是跳转到 复位向量。这里定义了当发生中断、复位、未定义指令等异常时，CPU 该去哪里执行代码
    2. 设置 CPU 为管理模式（SVC模式）
    作用：关中断，并将 CPU 设置为 SVC模式（特权模式），为后续操作准备好权限。
    3. 关闭看门狗、禁用中断
    作用：防止看门狗在 U-Boot 初始化过程中超时复位，同时屏蔽所有外设中断，避免干扰。
    4. 初始化关键硬件
    初始化系统时钟：设置 CPU 主频、总线频率。
    初始化 SDRAM/DDR：配置内存控制器，让 CPU 能访问内存。这是非常重要的一步，因为后续的代码和数据都要在内存中运行。
    设置栈指针（SP）：这是进入 C 语言世界的关键一步。有了栈，才能调用 C 函数、定义局部变量。
    5. 重定位
    如果 U-Boot 当前在 Flash 中运行，速度会比较慢。这一步会把 U-Boot 自身从 Flash 复制到内存（RAM） 中，然后跳转到内存中的地址继续执行，以提高运行速度
    6. 清BSS段
    作用：C 语言规定未初始化的全局变量值应为 0。这一步会把 BSS 段对应的内存区域全部清零。
    7. 跳转到 C 语言入口
    执行 bl board_init_f，进入第一个 C 函数。

第二阶段：C语言阶段（初始化与加载内核）
    1. 阶段一初始化：board_init_f
    运行位置：此时可能还在 Flash 或刚 relocate 到内存的低地址。

    主要任务：
    初始化一些基础外设：串口（这样就能看到打印信息了）、定时器等。
    规划内存布局：计算出代码段、数据段、堆、栈最终要放在内存的什么位置。这个结果会保存在一个全局结构体 gd_t 中。
    2.重定位（真正的搬移）
    作用：把 U-Boot 从 Flash 搬到 RAM 中。
    3.阶段二初始化：board_init_r
    运行位置：此时 U-Boot 已运行在 RAM 中。
    主要任务（这个函数里会调用一系列初始化函数，简称 init_sequence_r）：

    板级初始化：board_init（设置 GPIO、复用功能等）。
    中断初始化：interrupt_init、enable_interrupts。
    设备初始化：initr_serial（串口）、initr_net（网卡）、initr_mmc（SD卡）。
    环境变量加载：env_relocate（从存储设备加载环境变量）。
    命令行初始化：cli_init。
    4. 进入命令行或自动启动内核

5、有 cache 的 CPU 上使用 DMA 如何保证数据的一致性？
在 DMA 操作的关键时刻，让 CPU 主动去"刷新"或"失效" Cache，保证 Cache 和内存中的数据是一致的。
初学者容易犯的一个错误是：在启动 DMA 之前，只做了 Flush，没做 Invalidate；或者在 DMA 完成后，只做了 Invalidate，没做 Flush。
还有一种更隐蔽的错误：同时做了 Flush 和 Invalidate。

DMA 前：clean 或 flush（把脏数据写回内存）。
DMA 后：invalidate（丢弃 Cache 中的旧数据）。

6、阻塞休眠原理
关键特征：
不占 CPU：处于休眠状态的任务不会被调度器选中，不消耗 CPU 时间。
被唤醒后继续：当等待的事件发生时，任务会被内核唤醒，重新进入就绪队列，参与调度。
阻塞休眠的底层实现
1. 核心数据结构：等待队列（Wait Queue）
等待队列是阻塞休眠机制的核心数据结构。它本质上是一个链表，链表上挂着的都是正在等待某个事件发生的任务。
2. 阻塞休眠的过程：把自己挂到等待队列上
3. 唤醒的过程：把任务从等待队列移回就绪队列

7、tasklet 和 workqueue 的底层区别？
tasklet 运行在中断上下文，不能睡眠；Workqueue 运行在进程上下文，可以睡眠。
三、使用场景对比
场景	                用 Tasklet	                    用 Workqueue
中断处理下半部	        ✅ 非常适合	                    ❌ 太重量级
需要快速执行、不阻塞	✅ 首选	                        ⚠️ 也可以，但有额外开销

8、RTOS 采取哪些手段保证他的实时性？
1、可抢占式内核
2、任务调度策略：基于优先级抢占
3、中断处理：延迟尽可能短
4、中断处理的分层设计

9、RTOS 中哪里会进行线程调度
主动触发、被动触发、中断触发
1、主动触发：任务主动让出CPU 
2、被动出发：时间片轮转： Systick中断
3、中断触发：ISR唤醒高优先级任务


9、线程切换为什么用 PendSV，有什么优势？
PendSV 属于是一个异常，可以在高优先级中断正在处理时，还能安全、干净完成线程切换
如果没有 PendSV，线程切换会在中断处理过程中进行，这会导致中断处理过程中出现线程切换，从而影响实时性。
```C 
void SysTick_Handler(void) {
    // 1. 保存当前任务上下文
    // 2. 找到下一个要运行的任务
    // 3. 恢复新任务上下文
    // 4. 返回时直接跑到新任务
}
```
带来的问题：
1、中断延迟不可控
2、中断嵌套复杂化
3、抢占规则混乱


10、串口发送一个字节多少位
串口发送一个字节通常是 10 位或 11 位，具体取决于配置。它不仅仅是传输数据本身，还包含了起始位、停止位，以及可选的校验位。
