阻塞、非阻塞、POLL、SELECT、异步通知
1. 阻塞 I/O (Blocking I/O)
这是最直观的方式。你站在快递柜前，一直等着快递员来放包裹。在此期间，你什么也做不了。
工作流程：
你调用 read 系统调用。
内核开始等待数据准备好（比如等待网卡数据到来）。在此期间，你的进程被挂起（阻塞），不占用 CPU。
数据准备好后，内核将数据从内核空间拷贝到用户空间的缓冲区。这个拷贝过程，你的进程仍然在等待（阻塞）。
拷贝完成，read 调用返回，你的进程继续运行 。
特点：简单直接，但在等待期间 CPU 资源被浪费了。这是默认的 I/O 模式。

2. 非阻塞 I/O (Non-blocking I/O)
你变得很忙。不停地在快递柜和家之间来回跑，每次都去打开柜门看看包裹到了没有，没到就立刻回家干别的，过一会儿再来看看。
工作流程：
你将文件描述符设置为非阻塞模式，然后调用 read。
如果数据没有准备好，内核立即返回一个错误（如 EWOULDBLOCK），而不是让进程休眠。你的进程可以继续执行其他任务 。
你需要不断轮询（polling），重复调用 read 来检查数据是否准备好。
直到某次轮询，数据已准备好，此时 read 调用进入第二阶段，将数据从内核拷贝到用户空间。注意：这个拷贝过程是阻塞的！ 。
特点：进程不会被阻塞，但轮询会消耗大量的 CPU 时间，效率较低。

3. I/O 多路复用 (I/O Multiplexing) —— select 和 poll
你找了个帮手（select/poll），让他同时帮你盯着好几个快递柜。你只需要问这个帮手：“我关心的这几个柜子，有哪个到货了？”帮手检查后告诉你结果，你再根据结果去取。
工作流程：
你构建一个文件描述符（快递柜）的列表，然后调用 select 或 poll。
你的进程在调用 select/poll 时被阻塞。但内核现在帮你监控列表里的所有描述符 。
当任何一个描述符准备好数据（或有事件发生），select/poll 返回。
你遍历返回的结果，找到那些“就绪”的描述符，然后调用 read 去读取数据。注意：这个 read 调用虽然是阻塞的，但因为数据已经就绪，所以通常会很快返回 。
select 的特点：
文件描述符数量限制：通常由 FD_SETSIZE 宏限制，默认为 1024 。
效率问题：每次调用都需要将整个文件描述符集合从用户态拷贝到内核态，并且内核需要线性扫描整个集合来找出就绪的描述符。当描述符数量很大时，性能会线性下降 。
poll 的特点：
解决了数量限制：poll 使用链表结构存储描述符，因此没有最大数量的限制 。
但仍有性能问题：它和 select 一样，仍然需要将整个描述符列表拷贝到内核，并在返回后由用户程序遍历整个列表来查找就绪事件。在描述符数量巨大时，效率依然不高 。

4. 异步通知 (Asynchronous I/O - AIO)
这是最省心的方式。你给快递公司打了个电话，说“包裹到了直接帮我送上门，到了给我打个电话就行”。然后你就该干嘛干嘛去了。等包裹送到家并放好，你才会接到电话通知。
工作流程：
你调用 aio_read 等异步 I/O 函数，告诉内核你要读取哪个文件描述符，数据要拷贝到哪里，以及操作完成后如何通知你（例如通过信号或回调函数）。这个调用立刻返回，你的进程可以继续做其他事，完全不被阻塞 。
内核负责等待数据准备好。
内核还负责将数据从内核空间拷贝到你指定的用户空间缓冲区。
一切都完成后，内核通过你指定的方式（如信号）通知你，“数据已就位，请查收” 。
特点：这是最理想的 I/O 模型，进程在整个 I/O 操作过程中都无需等待。真正实现了“零阻塞”。
